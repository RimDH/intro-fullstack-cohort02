Git & GitHub Mastery

Version Control Workflow for Professional Developers

What is Git vs. GitHub?

Git (The Engine)

A Distributed Version Control System that tracks changes in your local files. It works offline.

Snapshots
Local

GitHub (The Cloud)

A cloud platform to host Git repositories. It enables collaboration, Code Review, and CI/CD.

Remote
Open Source

Key Terms

Repository: Your project folder. Commit: A saved snapshot. Branch: A parallel version of code.

01. Installation & Verification

Before coding, ensure Git is installed on your machine.

  1. Create Account: Sign up at github.com
  2. Download Git: Visit git-scm.com

Verify Installation

git --version

If you see a version number (e.g., 2.x.x), you are ready to go.

02. Global Git Configuration

Set your identity so your commits are attributed to you.

git config --global user.name "Your Full Name"
git config --global user.email "YourEmail@test.com"
git config --global init.defaultBranch main
git config --global core.editor "code --wait"

The last command ensures VS Code is your default editor for commit messages.

03. SSH + Secure Connection

Instead of typing your password every time, use SSH Keys for secure, automated authentication.

  1. Open Git Bash and generate key: 
    ssh-keygen -t ed25519 -C "Youremail@test.com"
  2. Start the agent and copy the key: 
    eval "$(ssh-agent -s)"
cat ~/.ssh/id_ed25519.pub

Connecting to GitHub

1. Copy the output from the previous slide starting with ssh-ed25519...

2. Go to GitHub: Settings -> SSH and GPG keys -> New SSH key.

Test Connection

ssh -T git@github.com

If successful, you'll see: "Hi [User]! You've successfully authenticated..."

04. Cloning & Initializing

Option A: Clone an existing repo

git clone git@github.com:USERNAME/proj_name.git

Option B: Initialize in an existing local folder

git init
git add .
git commit -m "Initial commit"
git remote add origin SSH_LINK_OF_YOUR_REPO
git push origin main

05. The "Big Three" Workflow

The daily cycle of a developer:

git add .               # Stage all changes
git commit -m "Comment" # Save snapshot locally
git push origin main    # Upload to the cloud

06. Feature Branching

Never work directly on main. Create a branch for new features.

git branch my-new-branch      # Create
git checkout my-new-branch    # Switch to it
# ... make changes ...
git push -u origin my-new-branch

Note: -u sets the "upstream" so you can just type git push next time.

07. Merging Local Updates

After your feature is done, bring those changes into main.

git checkout main         # Go to main
git pull origin main      # Get latest updates from others
git merge origin/my-new-branch
git push origin main

Repeat this for every branch to keep the project synchronized.

Syncing the Team

After a merge, every developer needs to update their local branch.

git pull origin main

What is "Fetch"?

git fetch origin downloads the latest history from GitHub but **does not touch your files**. It's safe to run anytime to "look" at what others have done.

08. Rollbacks: The Panic Buttons

Case 1: Merged locally but NOT pushed

If you made a mistake and haven't uploaded it yet:

git reset --hard HEAD~1

Warning: This deletes your local work in that commit!

09. Rollbacks: Reverting Public Code

Case 2: You already PUSHED the mistake

Never "Reset" a public branch. Use "Revert" to create a new commit that undoes the bad one.

git log --oneline              # Find the bad commit hash
git revert -m 1 <hash>          # Create a "Correction" commit

This keeps the history clean and safe for other teammates.

Pro Features to Know

  • Pull Requests (PR): Asking teammates to review code before merging on GitHub.
  • Git Lens: A tool (VS Code extension) to see who wrote which line.
  • .gitignore: A file that tells Git which files (like passwords) to ignore.
